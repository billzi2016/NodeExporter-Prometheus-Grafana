# Grafana PRD

## 1. 文档目的

本文档用于定义本项目中 `Grafana` 模块的产品需求、展示目标和验收标准。

在这套监控系统里，`Grafana` 的职责不是单纯“能打开页面”，而是把来自 Prometheus 的宿主机监控数据组织成清晰、可用、可扩展的可视化界面，让用户直接观察电脑当前与历史状态。

## 2. 背景与问题

当前 `API_test` 目录已经可以打印很多本机状态，但文本输出只适合手工检查，不适合长期观察、趋势分析和统一展示。

如果没有 Grafana：

- 无法看到资源变化趋势
- 无法把多个指标集中到一个界面
- 无法形成适合日常观察的监控面板

所以 Grafana 的角色是把“零散指标”变成“可持续使用的监控界面”。

## 3. 产品目标

- 提供一个 Docker 化部署的 Grafana 服务
- 接入 Prometheus 作为数据源
- 构建面向宿主机监控的基础仪表盘
- 让用户能直观看到电脑当前和历史的 CPU、内存、磁盘、网络等状态

## 4. 非目标

- 本阶段不追求复杂组织架构和企业级权限模型
- 本阶段不处理多租户场景
- 本阶段不优先构建大量高级面板模板市场能力
- 本阶段不做定制插件开发

## 5. 用户故事

### 5.1 作为本机使用者

我希望打开 Grafana 后能快速看到我的电脑资源状态，而不是先自己拼查询语句。

### 5.2 作为开发者

我希望 Grafana 能和 Docker 化的 Prometheus 顺畅联通，默认就能展示核心面板。

### 5.3 作为后续维护者

我希望后面扩展新的面板或数据源时，不需要重做整个可视化结构。

## 6. 功能需求

### 6.1 部署形态

- Grafana 必须以 Docker 容器形式运行
- 必须纳入统一编排
- 需要暴露本地访问端口

### 6.2 数据源

- 第一阶段必须接入 Prometheus
- 数据源配置需要可复用、可版本化
- 后续应保留接入其他数据源的可能性
- Grafana 不强依赖官方 `node exporter`，而是依赖最终那条能够正确表达宿主机状态的 Prometheus 数据链路

### 6.3 仪表盘目标

第一阶段需要至少覆盖以下宿主机维度：

- CPU 总体使用率
- 内存总量、使用量、剩余量
- 磁盘使用率、主要挂载点容量
- 网络收发趋势
- 主机运行时长、基础状态信息

### 6.4 页面体验

- 首页应以“宿主机当前状态总览”为中心
- 关键指标必须能在首屏快速识别
- 图表应兼顾当前值和趋势
- 面板结构应足够清晰，后续便于扩展 GPU 或自定义指标

### 6.5 可维护性

- 仪表盘配置应支持导出和版本管理
- 数据源和面板不应完全依赖手工点击创建
- 后续应能逐步过渡到配置化或 provisioning 模式

## 7. 约束与设计原则

### 7.1 先可用，再精致

第一阶段目标是形成稳定、清晰、可验证的宿主机监控面板，而不是先做复杂视觉包装。

### 7.2 宿主机导向

Grafana 展示的核心对象必须是这台电脑，而不是 Docker 容器内部资源。

### 7.2.1 上游实现可以替换，但展示目标不能变

即使后续为了满足 macOS 宿主机监控需求，把 `node` 这一层从官方 `node exporter` 切换为自定义 exporter，Grafana 的展示目标也不变：  
仍然必须围绕宿主机状态构建面板，而不是围绕容器内部状态构建面板。

### 7.3 面板少而关键

首版不追求堆很多图，而是优先保证每个图都对应一个真实使用场景。

### 7.4 为扩展留结构

后续可以逐步加入 GPU、温度、自定义脚本采样结果等内容，所以当前面板结构应保留扩展余地。

## 8. 交互与展示需求

后续实现阶段建议至少具备：

- 项目根目录必须统一放置 `docker-compose.yml`、`.dockerignore` 等编排级文件
- `Grafana` 必须有独立服务目录，例如 `grafana/`
- 一个总览 Dashboard
- 若干资源分类面板区块
- 清晰的标题、单位和时间范围选择
- 默认时间范围适合观察近实时状态
- 外部访问端口在绑定前必须先检查是否已被占用
- 如果默认端口已被占用，则按顺序顺延到下一个可用端口

## 9. 项目结构要求

本项目后续实现阶段至少应遵循以下目录边界：

```text
NodeExporter-Prometheus-Grafana/
├── docker-compose.yml
├── .dockerignore
├── .gitignore
├── node-exporter/
├── prometheus/
├── grafana/
├── API_test/
└── docs/
    └── prds/
```

目录职责说明：

- `node-exporter/`：放置 node exporter 相关配置、说明和后续可能的自定义内容
- `prometheus/`：放置 Prometheus 配置、规则、数据卷组织说明等内容
- `grafana/`：放置 Grafana provisioning、dashboard、datasource 和可视化相关文件
- `API_test/`：保留为实验与验证目录，不等同于正式监控服务实现
- `docs/prds/`：存放需求文档，作为后续实现和迭代依据

## 10. 风险与关注点

- 如果上游 Prometheus 指标结构不稳定，Grafana 面板会频繁返工
- 如果 `node exporter` 在容器化场景下没有正确反映宿主机数据，Grafana 层再漂亮也没有意义
- 如果最终切换到自定义 exporter，需要在 Dashboard 和 README 中清楚说明其指标来源
- 首期应该先用标准面板验证链路，避免过早为视觉效果牺牲配置清晰度

## 11. 验收标准

满足以下条件视为 Grafana 第一阶段完成：

- 可以通过 Docker 成功启动
- 服务目录结构符合项目约定
- 可以成功接入 Prometheus 数据源
- 可以看到宿主机 CPU、内存、磁盘、网络基础面板
- 面板数据与主机实际运行状态基本一致
- 仪表盘结构可以支持后续扩展
- 最终实际绑定端口经过占用检查，且能稳定访问

## 12. 后续扩展

- 增加 GPU 相关面板
- 增加业务 API 指标展示
- 增加告警可视化
- 把数据源和仪表盘切换为更完整的 provisioning 管理方式
