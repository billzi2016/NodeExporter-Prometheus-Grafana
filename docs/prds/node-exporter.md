# Node Exporter PRD

## 1. 文档目的

本文档用于定义本项目中 `node exporter` 模块的产品需求、部署边界和验收标准。

当前仓库中的 `API_test` 目录仅用于本机状态采样验证，不作为最终监控系统实现。正式系统将进入 Docker 化阶段，并由 `node exporter + Prometheus + Grafana` 组成完整链路。

本 PRD 的核心目标，是明确 `node exporter` 必须以容器形式运行，但采集对象必须是宿主机，而不是容器自身。

## 2. 背景与问题

当前已经有一组本地 Python 测试脚本，可以观察 CPU、内存、GPU、网络、硬盘等信息，但这些脚本的定位更偏向手工验证和原型探索，不能直接作为标准化监控系统使用。

要把这套能力产品化，必须引入标准监控链路：

- `node exporter` 负责暴露宿主机资源指标
- `Prometheus` 负责拉取和存储时间序列数据
- `Grafana` 负责可视化展示和后续扩展

问题的关键不在于是否容器化，而在于容器化后不能只看到容器自己的隔离资源视图。如果 `node exporter` 只采集容器自身信息，那么结果对“监控电脑当前状态”这个目标没有意义。

## 3. 产品目标

- 提供一个 Docker 化部署的 `node exporter`
- 暴露宿主机 CPU、内存、磁盘、文件系统、网络等基础指标
- 供 `Prometheus` 稳定抓取
- 为 `Grafana` 提供宿主机级监控数据源
- 为后续引入自定义 exporter 或业务采样模块保留扩展空间

## 3.1 技术路线优先级

本项目在 `node` 这一层采用明确的优先级策略：

1. 优先尝试纯官方方案  
   即优先使用官方 `node exporter` 容器，尽量不自定义实现。
2. 如果纯官方方案不能满足“监控宿主机而不是容器”的核心目标  
   则切换为自定义 exporter 方案。
3. 自定义 exporter 方案的基础来源  
   优先复用当前 `API_test` 中已经验证过的本机采样能力，并通过 `FastAPI` 或等价方式暴露给 Prometheus。

这条优先级策略必须贯穿实现、文档和最终交付说明。

## 4. 非目标

- 不在本阶段处理告警逻辑
- 不在本阶段处理多节点集群
- 不在本阶段处理复杂鉴权、TLS、外网暴露
- 不在本阶段让 `node exporter` 承担业务数据采集职责
- 不以 `API_test` 脚本替代 `node exporter`

## 5. 用户故事

### 5.1 作为开发者

我希望通过一个 Docker 容器运行 `node exporter`，这样部署方式统一、后续可以直接和 Prometheus/Grafana 一起编排。

### 5.2 作为本机使用者

我希望在 Grafana 里看到的是这台电脑本身的 CPU、内存、硬盘、网络状态，而不是某个容器自己的视图。

### 5.3 作为后续维护者

我希望 `node exporter` 的职责边界清晰，只负责稳定暴露宿主机基础指标，不把定制业务采样混在里面。

## 6. 功能需求

### 6.1 部署形态

- `node exporter` 必须以 Docker 容器形式运行
- 容器必须纳入项目统一编排
- 容器配置必须允许访问宿主机需要采集的系统信息

### 6.2 采集目标

必须覆盖以下宿主机资源维度：

- CPU 使用率
- 内存总量、已用、可用、缓存等信息
- 磁盘容量、文件系统使用率、挂载点
- 网络接口、收发流量、错误计数
- 主机基础运行指标，例如 load、boot time、上下文切换等标准 node exporter 指标

### 6.3 指标暴露

- 必须暴露 Prometheus 可抓取的 `/metrics` 接口
- 接口输出格式必须兼容 Prometheus 标准文本格式
- 指标名称优先遵循官方 `node exporter` 既有规范

### 6.4 宿主机可见性

这是本模块最重要的约束：

- 指标来源必须是宿主机
- 不能仅反映容器内部命名空间看到的资源
- 需要通过挂载宿主机路径、指定 rootfs 路径、必要权限等方式实现宿主机采集
- 文档和实现都必须明确说明“监控对象是宿主机”

### 6.5 运行可观测性

- 容器启动后必须能被健康检查
- 当宿主机路径挂载异常或权限不足时，应有可诊断的启动日志

## 7. 约束与设计原则

### 7.1 容器化优先

即使本机直接运行 exporter 更简单，本项目也仍然以 Docker 化为统一部署方式，避免后续环境分裂。

### 7.2 宿主机视角优先

所有设计必须服从“看电脑，不看容器”这个原则。任何让指标退化为容器内部视角的方案都不应接受。

### 7.3 尽量复用官方能力

优先使用官方 `node exporter` 提供的标准能力，不要过早引入大量自定义采集逻辑。

### 7.4 与 API_test 解耦

`API_test` 只保留为实验和对照工具，不直接作为正式监控数据链路的一部分。

### 7.5 官方失败时允许进入自定义实现

如果官方 `node exporter` 在目标环境下无法稳定提供宿主机视角指标，则允许进入自定义实现路线。  
这里的“允许”不是随意偏离标准，而是受明确条件触发的 fallback 策略。

## 8. 部署需求

后续实现阶段需要满足以下要求：

- 项目根目录必须统一放置 `docker-compose.yml`、`.dockerignore` 等编排级文件
- `node exporter` 必须有独立服务目录，例如 `node-exporter/`
- 以 Docker 容器运行 `node exporter`
- 通过挂载宿主机必要路径获取宿主机文件系统和系统状态
- 必要时通过 rootfs 参数让 exporter 以宿主机视角读取指标
- 默认外部暴露端口从 `9100` 开始
- 在真正绑定宿主机端口前，必须先检查该端口是否已被占用
- 如果 `9100` 已被占用，则按顺序顺延到下一个可用端口，例如 `9101`、`9102`
- 保证配置方式可以稳定写入 `docker-compose.yml`

## 8.1 官方方案验证结论

在当前项目环境下，已经做过一次纯官方验证：

- Docker 可以启动官方 `prom/node-exporter`
- 容器可以正常暴露 `/metrics`
- 但暴露出来的关键系统信息表现为 Linux / LinuxKit / 容器视角，而不是 macOS 宿主机视角

这说明“官方方案可运行”不等于“官方方案满足需求”。  
对于本项目而言，判断标准不是容器有没有启动成功，而是指标是否真正代表这台电脑本身。

已观察到的典型现象包括：

- `node_uname_info` 显示的是 Linux 内核信息
- 节点名表现为容器标识，而不是宿主机标识
- 指标语义更接近 Docker Desktop 背后的 Linux 环境，而不是 macOS 本机

因此，在当前 macOS + Docker Desktop 场景中，纯官方方案存在较大概率不能直接满足核心目标。

## 8.2 自定义 Fallback 方案

如果后续进一步验证仍然证明官方 `node exporter` 不能提供宿主机级指标，则进入 fallback 方案：

- 保留 `Prometheus` 与 `Grafana` 继续使用官方组件
- `node` 这一层改为自定义 exporter
- 采样逻辑优先复用 `API_test` 中已有的 CPU、内存、GPU、网络、硬盘采样能力
- 对外通过 `FastAPI` 或等价方式暴露接口
- 再由 Prometheus 抓取 Prometheus 兼容指标

这里要特别强调：

- 目标不是“为了自定义而自定义”
- 目标是确保最终看到的是宿主机状态
- 只有当官方方案在目标环境里不能满足这个目标时，才进入该路线

## 9. 项目结构要求

本项目后续实现阶段至少应遵循以下目录边界：

```text
NodeExporter-Prometheus-Grafana/
├── docker-compose.yml
├── .dockerignore
├── .gitignore
├── node-exporter/
├── prometheus/
├── grafana/
├── API_test/
└── docs/
    └── prds/
```

目录职责说明：

- `node-exporter/`：放置 node exporter 相关配置、说明和后续可能的自定义内容
- 如果最终无法使用纯官方 `node exporter`，该目录需要容纳自定义 exporter 的实现与部署文件
- `prometheus/`：放置 Prometheus 配置文件、规则文件和数据组织说明
- `grafana/`：放置 Grafana provisioning、dashboard、datasource 等文件
- `API_test/`：保留为本机采样验证与实验目录，不作为正式监控服务目录
- `docs/prds/`：存放 SDD / PRD 文档，先定义需求，再推动实现

## 10. 风险与关注点

- Docker Desktop / macOS 场景下，宿主机指标可见性与 Linux 原生环境不同，后续实现必须验证可行边界
- 某些宿主机指标在容器化场景下可能存在缺失或语义偏差，需要在实现和验收阶段明确
- 如果后续发现官方 `node exporter` 对 macOS 宿主机容器化观测存在天然限制，需要及时记录并评估是否引入补充 exporter
- 如果最终放弃纯官方 node exporter，必须在项目 `README` 中明确记录原因、验证结果和替代方案

## 11. 验收标准

满足以下条件视为本模块达到第一阶段目标：

- 可以通过 Docker 启动 `node exporter`
- 服务目录结构符合项目约定
- `Prometheus` 可以成功抓取其 `/metrics`
- 至少能看到宿主机级别的 CPU、内存、磁盘、网络基础指标
- 采集结果应与本机实际状态大体一致，而不是仅表现为容器资源
- 配置方式可被纳入项目统一编排
- 最终实际绑定端口经过占用检查，且能稳定提供服务
- 如果最终采用自定义 exporter，则必须在 `README` 中清楚说明未使用纯官方方案的原因

## 12. 后续扩展

- 如官方指标不足，可补充自定义 exporter
- 可逐步纳入温度、电源、GPU 等更细粒度监控
- 可在后续版本中引入告警规则和多机支持
